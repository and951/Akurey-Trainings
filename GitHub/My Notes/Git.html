¿Note cómo en la sección pasada Git dice octocat.txt es "Untracked"? Eso significa que Git ve que octocat.txt es un archivo nuevo.
Para decirle a Git que inicie el “tracking" de los cambios realizados en octocat.txt, lo primero que tenemos añadir que el área mediante el uso de git add. En el caso anterior seria util usar :

git add octocat.txt

Git branching Model

En general se manejan los repositorios como uno principal o origin y de esta todos los miembros realizan sus acciones a partir de esta.



De este repositorio salen ciertas branchs principales:

  • Master: Este es aquel que lleva la versión del proyecto con una etapa lista en producción.
  • Develop:  Este branch es aquel donde se refleja un estado con los últimos cambios de desarrollo próximos de liberar a producción. 


Es importante resaltar que estas branchs tienen una vida valida del tamaño del mismo proyecto.

Por otra parte tenemos otras branchs de soporte que ayudan al desarrollo paralelo entre los miembros del equipo, facilita el seguimiento de características, preparar para los lanzamientos de producción y para ayudar a fijar rápidamente problemas de producción en directo. Al contrario de las branchs anteriores estas si tienen un tiempo de vida definido.

Feature branchs son utilizadas sobre todo para cuando se debe de desarrollar una nueva característica para el siguiente release.La esencia del feature branch es que existe siempre y cuando la característica está en desarrollo, pero eventualmente se hará un merge con develop.

Crear un un feature branch

$ git checkout -b myfeature develop
Switched to a new branch “myfeature”

Incorporar el feature branch al develop branch

$ git checkout develop
Switched to branch 'develop'
$ git merge --no-ff myfeature
Updating ea1b82a..05e9557
(Summary of changes)
$ git branch -d myfeature
Deleted branch myfeature (was 05e9557).
$ git push origin develop



La bandera —no-ff causa que el merge siempre crea un commit object aunque pueda realizar un fast forward.  Esto evita perder información sobre la existencia histórica de un feature branch  y agrupa a todos los commits que había añadido a la función.

Los release branches se utilizan para todos los preparativos para sacar una versión a producción. Además se utilizan para arreglar bugs menores y preparar el meta-data para la salida a producción. Haciendo esto se despeja el develop branch y lo deja listo para la siguiente iteración y las cosas que se deben de realizar.

Crear un release  branch
$ git checkout -b release-1.2 develop
Switched to a new branch "release-1.2”
$ ./bump-version.sh 1.2
Files modified successfully, version bumped to 1.2.
$ git commit -a -m "Bumped version number to 1.2”
[release-1.2 74d9424] Bumped version number to 1.2
1 files changed, 1 insertions(+), 1 deletions(-)

Incorporar el release branch al develop branch

$ git checkout master
Switched to branch 'master'
$ git merge --no-ff release-1.2
Merge made by recursive.
(Summary of changes)
$ git tag -a 1.2

$ git checkout develop
Switched to branch 'develop'
$ git merge --no-ff release-1.2
Merge made by recursive.
(Summary of changes)


Cuando se deja de utilizar el release branch se debe de eliminar.

$ git branch -d release-1.2
Deleted branch release-1.2 (was ff452fe).

 El hot fix branch se utiliza cuando surge un bug severo que debe de ser arreglado con urgencia. Estas surgen de cierto tag de el master que marca algo en producción.



Su esencia es que los demás puedan seguir trabajando en sacar el sistema a producción mientras alguno se encarga de arreglar el problema encontrado.

Crear un hotfix branch

$ git checkout -b hotfix-1.2.1 master
Switched to a new branch "hotfix-1.2.1"
$ ./bump-version.sh 1.2.1
Files modified successfully, version bumped to 1.2.1.
$ git commit -a -m "Bumped version number to 1.2.1"
[hotfix-1.2.1 41e61bb] Bumped version number to 1.2.1
1 files changed, 1 insertions(+), 1 deletions(-)
Cuando se arregla el problema 

$ git commit -m "Fixed severe production problem"
[hotfix-1.2.1 abbe5d6] Fixed severe production problem
5 files changed, 32 insertions(+), 17 deletions(-)


Finalizar un hotfix branch

$ git checkout master
Switched to branch 'master'
$ git merge --no-ff hotfix-1.2.1
Merge made by recursive.
(Summary of changes)
$ git tag -a 1.2.1

Luego se hace merge a develop en caso de que no haya un release branch, de ser así se le hace merge al release branch.

 $ git checkout develop
Switched to branch 'develop'
$ git merge --no-ff hotfix-1.2.1
Merge made by recursive.
(Summary of changes) 

Por ultimo cuando ya este todo arreglado se debe de eliminar el branch.
