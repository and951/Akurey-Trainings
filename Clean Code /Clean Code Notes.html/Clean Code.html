Formato
El formato es de las primeras impresiones que se da acerca del proyecto hacia aquellos que lo supervisan. Por dicha razón si el código tiene un buen formato se empezara con confianza la revisión. Por otro lado si el código posee un formato desordenado y sin cuidado la primera impresión es que todo el proyecto tiene esa misma calidad.

Se mencionan ciertas cualidades que tiene un proyecto con un buen formato.

  1. Los archivos deben de ser cortos. Hacer la analogía de un proyecto como un periódico. Y cada archivo debe de ser como un articulo. Donde el encabezado es el nombre del archivo y debe de describir lo que se realiza en el archivo. 
  2. Apertura Vertical. Es importante que en un archivo cada sección, llámese la declaración de atributos en una clase, donde se importan librerías necesarias o entre los métodos este separado por un espacio en blanco. Esto para poder facilitar la lectura y la separación de ideas a primer vista.
  3. Densidad Vertical. Es importante que los archivos relacionados estén cerca uno del otro y que no se usen comentarios innecesarios que rompan este esquema

Clases

Para lo que son las clases se sigue el estándar de Java. Este estándar especifica que los atributos deben estar declarados al principio seguido por las clases. Pero de igual manera que las funciones se debe de seguir el principio de SRP (Single Responsability Principle). Osea que las clases deben de ser lo mas cortas posibles, y se debe de calcular dicho tamaño con las responsabilidades asignadas. 

Otro aspecto importante que se debe tener en cuenta cuando se construye un sistema orientado a objetos es la cohesión. La cohesión se define como el nivel de dependencia entre los elementos presentes. Una buena practica es que haya buena cohesion en las clases. Esto permite al identificar los elementos que no cumplan este principio, como elementos que no pertenecen a la clase.


ADT (Abstract Data Types)

Es importante el uso de los tipos abstractos en la programación orientada a objetos. Su uso permite un verdadero sistema modular donde se pueden definir partes del sistema que al cambiarse no afecte al resto.

Además trae otros beneficios como:
 
  1. El sistema es más facil de controlar en cuestiones de eficiencia.
  2. El sistema se llega a auto documentar.
  3. Evita el uso de variables globales, o que se tengan que pasar información entre el sistema.
  4. Permite que se trabaje con entidades más que con estructuras implementadas en bajo nivel.

Encapsulamiento

El encapsulamiento es también una parte fundamental ya que sin este por lo general se quiera el modelo abstraído. Por lo general se debe de seguir las siguientes reglas para tener un buen nivel de encapsulamiento.

  1. No se debe de hacer públicos los atributos.
  2. No poner referencias de implementaciones privadas en las interfaces.
  3. Estar atento a revisar el encapsulamiento tanto de manera sintáctica como de manera semántica. A lo que se refiere es no solo dejar el encapsulamiento con que los atributos sean privados y no intentar accederlos, si no también ir  más allá en la definición y por ejemplo evitar hacer cosas como las siguientes que violan los principios de encapsulamiento.

    1. No llama a la rutina Initialize () de la clase A porque sabe que la rutina PerformFirstOperation () de la clase A lo llama automáticamente.
    2. No llamar a la rutina database.Connect () antes de llamar a employee.Retrieve (base de datos) porque sabe que la función employee.Retrieve () se conectará a la base de datos si aún no existe una conexión.
    3. No llama a la rutina Terminate () de la clase A porque sabe que la rutina PerformFinalOperation () de la clase A ya la ha llamado.
    4. Utilizar un puntero o una referencia a ObjectB creado por ObjectA incluso después de ObjectA ha salido del ámbito, porque sabe que ObjectA mantiene ObjectB en almacenamiento estático y ObjectB seguirá siendo válido.
    5. Utilizar la constante MAXIMUM_ELEMENTS de ClassB en lugar de utilizar ClassA.MAXIMUM_ELEMENTS, porque sabe que ambos son iguales al mismo valor.

Crear una rutina puede ser justificada no solo para evitar la duplicación de código. Se mencionan algunas razones justificables para crear una rutina:

  1. Evite el código duplicado
    Sin lugar a dudas, la razón más popular para crear una rutina es evitar el código duplicado. De hecho, la creación de código similar en dos rutinas implica un error en la descomposición.
  2. Ocultar secuencias
    Es una buena idea ocultar el orden en el que los eventos pasan a ser procesados. Por ejemplo, si el programa normalmente obtiene datos del usuario y luego obtiene datos auxiliares de un archivo, ni la rutina que obtiene los datos del usuario ni la rutina que obtiene los datos del archivo debe depender de que la otra rutina se realice primero.
  3. Ocultar operaciones de puntero
    Las operaciones de puntero tienden a ser difíciles de leer y propensas a errores. Al aislarlos en las rutinas (o en una clase, si es apropiado), puede concentrarse en la intención de la operación en lugar de en la mecánica de la manipulación del puntero. Además, si las operaciones se realizan en un solo lugar, puede estar más seguro de que el código es correcto. 
  4. Reducir la complejidad
    La razón más importante para crear una rutina es reducir la complejidad de un programa. Cree una rutina para ocultar la información de modo que no tenga que pensar en ello. 
  5. Hacer una sección de código legible
    Poner una sección de código en una rutina bien nombrada es una de las mejores maneras de documentar su propósito.
  6. Mejorar la portabilidad
    El uso de rutinas aísla las capacidades no portátiles, identificando explícitamente y aislando el trabajo de portabilidad en el futuro. L
  7. Simplifique las pruebas booleanas complicadas
    Rara vez es necesario comprender las pruebas booleanas complejas para comprender el flujo del programa. 
  8. Mejorar el rendimiento
    Puede optimizar el código en un lugar en lugar de varios lugares. Tener código en un lugar significa que una única optimización beneficia a todas las rutinas que utilizan esa rutina, ya sea que la utilicen directa o indirectamente.


Programación a la defensiva


Es importante a la hora de desarrollar un sistema tener en cuenta ciertos eventos que puedan afectar el sistema y de esa manera “curarse en salud”. Como por ejemplo proteger y hacer una buena validación de las entradas a recibir el sistema.

  1. Compruebe los valores de todos los datos de fuentes externas
  2. Compruebe los valores de todos los parámetros de entrada de rutina
  3. Decidir cómo manejar entradas incorrectas

Ademas para esto se puede utilizar los assertions o aserciones. Las aserciones se pueden usar para verificar suposiciones como estas:
 
  1. Que el valor de un parámetro de entrada cae dentro de su rango esperado (o el valor de un parámetro de salida sí)
  2. Que un archivo o flujo está abierto (o cerrado) cuando una rutina comienza a ejecutarse (o cuando termina de ejecutarse)
  3. Que un archivo o secuencia está al principio (o al final) cuando comienza una rutina (o cuando termina de ejecutarse)
  4. Que un archivo o una secuencia está abierto para sólo lectura, escritura o lectura y escritura
  5. Que el valor de una variable sólo de entrada no se cambia por una rutina
  6. Que un puntero no es NULL
  7. Que un array u otro contenedor pasado a una rutina puede contener al menos X número de elementos de datos
  8. Que una tabla ha sido inicializada para contener valores reales
  9. Que un contenedor está vacío (o lleno) cuando una rutina comienza a ejecutarse (o cuando termina)

Las aserciones se utilizan para manejar errores que nunca deberían ocurrir en el código. Por otro lado ¿Cómo manejas los errores que deberían ocurrir?

¿Cuánto tiempo debe ser un bucle?
La longitud del bucle se puede medir en líneas de código o profundidad de anidamiento. Aquí hay algunas pautas:
  1. Haga que sus bucles sean lo suficientemente cortos para verlo todo a la vez.
  2. Limitar el anidamiento a tres niveles Los estudios han demostrado que la capacidad de los programadores para comprender un bucle se deteriora significativamente más allá de tres niveles de anidación (Yourdon 1986a). Si va más allá de ese número de niveles, haga que el bucle sea más corto (conceptualmente) al romper parte de él en una rutina o simplificar la estructura de control.
  3. Mueva los bucles de bucles largos en rutinas Si el bucle está bien diseñado, el código en el interior de un bucle a menudo se puede mover a una o más rutinas que se llaman desde dentro del bucle.
  4. Haga bucles largos especialmente claros La longitud añade complejidad.   • Comience con un caso.
